COMBINED PRE-FEASIBILITY REPORT
Scope: single report containing (1) decision matrix, (2) 14-day MVP guardrail list, (3) post-MVP phase gates.
Constraint: theoretical planning only, no SSOT updates.

SECTION 1 - GO/NO-GO/HYBRID DECISION MATRIX
Objective: decide whether to refactor now or after MVP while preserving 14-day installer + beta readiness.

Scoring scale:
- 1 = poor/high risk, 5 = strong/low risk

Criteria (weighted):
1) 14-day release probability (weight 40)
2) Regression/control risk during release window (weight 25)
3) Long-term platform value (weight 20)
4) Decision reversibility and execution flexibility (weight 15)

Option A: Refactor now
- Score: release 1, regression 2, platform value 5, reversibility 2
- Weighted score: (1*40)+(2*25)+(5*20)+(2*15)=220/500
- Verdict: NO-GO if 14-day target is strict

Option B: Post-MVP refactor
- Score: release 4, regression 4, platform value 4, reversibility 4
- Weighted score: (4*40)+(4*25)+(4*20)+(4*15)=400/500
- Verdict: GO baseline

Option C: Hybrid (recommended)
- Definition: no full refactor now; apply strict pre-MVP guardrails, then phased post-MVP refactor.
- Score: release 4, regression 4, platform value 5, reversibility 5
- Weighted score: (4*40)+(4*25)+(5*20)+(5*15)=435/500
- Verdict: RECOMMENDED GO

Decision statement:
- For a first SaaS product with a hard 14-day packaging/distribution target, Hybrid is the highest-confidence path.

SECTION 2 - 14-DAY MVP GUARDRAIL TASK LIST
Goal: ship installer + beta-ready build without deep architectural refactor.

Guardrail principles:
1) No new provider-specific branching in core runtime path.
2) No expansion of LLM-specific core contracts.
3) No broad refactor touching pipeline contracts unless blocker-level bug.

Execution backlog (Day 1-14):
A) Release hardening and packaging
1) Freeze scope to release-critical items only.
2) Installer pipeline dry-runs and rollback rehearsal.
3) Smoke test matrix: start/stop, dispatch, connection lifecycle, usage view, error contract.

B) Stability controls
1) Add/keep regression checks for existing execution path (legacy + v1 surfaces currently in use).
2) Preserve deterministic logging and error payload contract.
3) Validate runtime process lifecycle and stop behavior on Windows.

C) Guardrail enforcement checks
1) Static check: no new provider-specific 'if provider == ...' in core execution modules.
2) Static check: no new hardcoded model/vendor strings in core.
3) Confirm dispatcher/UI changes do not introduce provider direct-calls.

D) Release gate
1) Installer generated and install/uninstall verified.
2) Beta checklist complete (known issues + support/runbook).
3) Go/No-Go review with release metrics and rollback readiness.

SECTION 3 - POST-MVP REFACTOR PHASE ENTRY/EXIT CRITERIA
Goal: convert toward policy-aware programmable execution fabric with controlled risk.

Phase A: Architectural baseline and contracts
- Entry:
1) MVP shipped.
2) First-week beta feedback triaged.
3) Refactor capacity reserved.
- Exit:
1) Core contract draft approved: RequestEnvelope, ExecutionResult, EventEnvelope.
2) Error taxonomy and compatibility rules documented.
3) No production behavior change yet.

Phase B: Compatibility envelope layer
- Entry:
1) Phase A contracts frozen.
2) Test plan for backward compatibility approved.
- Exit:
1) Compatibility adapter active (existing payloads still work).
2) Dual-path tests pass for old and new contract shapes.
3) No release regressions in current MVP flow.

Phase C: Hook chain foundation
- Entry:
1) Envelope layer stable.
2) Clear hook order approved (pre_context -> policy -> routing -> execution -> post).
- Exit:
1) Hook registry implemented with deterministic no-op defaults.
2) Router remains deterministic with explicit inputs/outputs.
3) Observability emits stage events without changing functional outcomes.

Phase D: Policy module migration
- Entry:
1) Hook chain stable in non-enforcing mode.
2) Persona/budget/interceptor migration plan approved.
- Exit:
1) Persona path runs via hook mechanism.
2) Budget/interceptor support monitor-only and enforce modes with explicit flags.
3) Determinism/regression suite green.

Phase E: Provider boundary cleanup
- Entry:
1) Policy hooks validated.
2) Provider metadata contract approved.
- Exit:
1) Provider-specific schema/runtime branches removed from core.
2) Adapter metadata owns provider-specific requirements.
3) Core execution path passes provider-agnostic compliance checks.

Phase F: Observability generalization and hardening
- Entry:
1) Core/provider boundaries stabilized.
2) Migration impact assessed for dashboards and exports.
- Exit:
1) Event/capability-oriented metrics coexist with legacy fields during transition.
2) Regression trend remains within accepted threshold.
3) Deprecation plan for legacy assumptions approved.

Governance checkpoints (all phases):
1) Each phase requires explicit entry/exit evidence.
2) No silent behavior drift: backward compatibility must be measurable.
3) If release-critical defects rise, pause refactor and return to stabilization mode.

FINAL RECOMMENDATION
- Choose Hybrid.
- Protect 14-day MVP target with strict guardrails.
- Start phased refactor immediately after MVP with explicit phase gates and compatibility controls.
